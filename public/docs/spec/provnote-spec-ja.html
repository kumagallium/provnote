1. provnote とは何か

provnote = Zettelkasten + オプショナル PROV。

provnote は、BlockNote.js ベースのブロックエディタの上に2つの仕組みを載せたノートアプリである。

Zettelkasten 基盤: あらゆるノートのあらゆるスコープ（見出し配下のブロック群）は、@ 参照で相互にリンクできる「Zettel（情報カード）」として機能する。ラベルの有無に関係なく、全ノート・全スコープがこの基盤の上にある。

オプショナル PROV: # ラベルを付けたスコープだけが、W3C PROV-DM の来歴グラフに登場する。ラベルを付けなければ PROV には一切関与しない。PROV は基盤の上に「乗っかる」追加機能であり、必須ではない。

これにより、雑多なメモも、実験記録も、論文の読書メモも、すべて同じエディタで扱える。ラベルを付けた分だけ構造化の恩恵を受けられるが、付けなくても Zettel としてネットワークに参加できる。

MOC (Map of Contents) は特別なノート種別ではない。 @ 参照が密集したノートが自然に MOC として機能する。ノートの「種類」という概念はなく、ラベル密度のグラデーションだけが存在する。

Google Drive 統合

ProvNoteDocument は JSON 文字列として Google Drive のファイルに保存される。MIME タイプは application/json、拡張子は .provnote.json。

  • 認証: OAuth 2.0 (Google Identity Services)。スコープは drive.file（アプリが作成したファイルのみアクセス可能）。
  • 保存: ファイル内容の更新は Drive API の files.update。自動保存（debounce 付き）。
  • 読み込み: files.get + alt=media で JSON を取得し、ProvNoteDocument にパース。
  • 一覧: files.list で .provnote.json ファイルを取得、左パネルに表示。
  • ノート間リンクの解決: BlockLink.targetNoteId は Google Drive のファイル ID。リンク先の読み込みは遅延（クリック時）。

3. 実験科学のドメイン知識

provnote は実験科学者の日常的なワークフローを主なターゲットにしている。以下の背景知識が設計判断の原地になっている。

実験ノートとは何か

実験科学（化学、材料科学、生物学など）では、研究者は「実験ノート」に手順・条件・結果を記録する。紙のノートが伝統的だが、近年は電子実験ノート (ELN: Electronic Lab Notebook) が使われる。法的・倫理的に「いつ・誰が・何をしたか」を証明できる記録が求められる。

provnote が目指すのは、ELN の堅牢性と、自由なメモ帳の気軽さの両立。既存の ELN はフォーム入力が多く、構造化を強制するため、研究者に嫌われがちで、結局紙のノートに戻るケースが多い。

典型的な実験ワークフロー

用語集

用語意味provnote での対応
試料 (Sample)実験の対象物。材料科学なら金属片、化学なら溶液、生物学なら細胞培養など。同じ手順を複数の試料に並行して適用し、条件を変えて比較するのが一般的。[試料] テーブル → 分割定義
条件手順における変数。温度、圧力、時間、濃度など。同じ手順でも試料ごとに異なる条件を設定する。テーブルの列 or [属性]

具体例: Cu焼結体の温度依存性

実験の概要

研究者が「Cu粉末を800℃, 900℃, 1000℃の3条件で焼結して、密度と硬度を比較したい」と考えた。手順は共通（焼結→アニール→評価）だが、温度だけが試料ごとに異なる。結果は3試料それぞれについて記録する。

途中脱落のケース

焼結中に S-C（1000℃の試料）が割れた場合、アニール以降では S-C は中断。だが、なぜ中断したかの記録として残す必要がある。タブを削除するのではなくスキップマークを付ける設計はこのため。

4. 設計原則

書く面は速く、読む面は賢く。 エディタは普通のメモ帳のまま。ユーザーの行為（ラベル付与、テーブル作成など）が機能を「召喚」する。何も書かなければ何も出ない。

ブロックは素材、ラベルは意味。 ラベルはブロック型と独立。H2 でも段落でもテーブルでも、同じラベルを付けられる。ブロック型は「素材」、ラベルは「意味」、PROV は「結果」。

ユーザーの行為が UI を召喚する。 ラベルを付けなければ PROV は生まれない。[試料] テーブルを書かなければタブは出ない。2つ目のタブを書き始めなければスキャフォールドは出ない。

5. ドメイン前提

以下は実験科学の現場の実感から導かれた前提であり、設計判断の原地。

前提1: 比較したいのは条件と結果、記録は試料ごとに分かれる

研究者が試料を並べて比較したいのは「どの条件でどんな結果が出たか」。これはテーブルやチャートで表現できる。一方、各試料の詳細な記録（SEM画像の観察メモ、手順中のトラブル記録など）は試料ごとに個別に書ければよい。横並びレーンではなくタブ方式が適切な根拠。構造化データの比較テーブルやチャートはタブの外（共通スコープ）に置くのが自然。

前提2: 記録は試料ごとに分かれていることが多い

典型的には1つずつ順番に焼結して、1つずつ結果を測定して、1つずつ記録する。ただし複数試料を同時に処理できる測定装置もある（多チャンネル測定器、バッチ引張処理等）。その場合でも記録は試料ごとに分かれていることが多い。全試料共通の操作メモ（「全部同じ炉で同時焼結した」等）はタブの外に書けばよい。

前提3: 比較には構造化データとリッチデータの2種類がある

構造化データの比較（数値結果）はテーブルの列とチャート（棒グラフ、散布図等）で完結する。これはタブの外（共通スコープ部分）に置くのが自然。リッチデータの比較（SEM画像の横並び等）はテーブルには入らず、比較ビュー（タブゾーン内の2分割表示）で対応する。

前提4: 試料セットは実験の最初に決まる

研究者は実験の最初に「今回の試料は S-A, S-B, S-C の3つ」と計画する。全手順を通じて共通で、手順ごとに試料が増えることは基本的にない（脱落はある、追加はない）。試料定義をノート冒頭に1回置き、全手順に自動適用する設計の根拠。

前提5: key:value の入力を強制できない

実験の途中で手が汚れていたり、急いでメモしたいこともある。フォーム入力を強制するとノートとして使われなくなる。漸進的構造化モデル（テキストのままでも動く → テーブルで書けば構造化される）の根拠。構造化の提案はAIチャットの中でのみ行い、エディタ上では割り込まない。

前提6: 試料間で key が揃う価値はあるが、強制はできない

全試料で「密度」「硬度」を持っていれば比較テーブルやチャートを自動生成できる。しかし、S-A は密度を測ったが S-B は測れなかった、は現実に起きる。スキャフォールド（最初のタブの構造を他タブにプレースホルダー表示）と補完性インジケーター（欠けの可視化）で動機づけはするが、強制はしない。

6. コアコンセプト: スコープ・Zettel・MOC

スコープ

provnote の全機能はスコープを単位に動作する。スコープ = 見出しブロック（H2, H3）+ 配下のブロック群。ノート全体も1つのスコープ。

スコープに対する3つの操作:

  • 入れ子 — H2 の中に H3。見出し階層。
  • 分割 — 試料タブ / ノート派生。並行する複数の記録。
  • 接続 — PROV リンク / @ 参照。スコープ間の関係。

スコープ = Zettel

あらゆるスコープは @ 参照で指せる。つまり、すべてのスコープが Zettelkasten における Zettel（情報カード）として機能する。

@ 参照は階層的なパスで細かい粒度まで指定できる:

@ノート名                       → ノート全体
@ノート名/焼結                  → H2 スコープ
@ノート名/焼結/S-A              → 試料タブ内
@ノート名/焼結/S-A/密度         → 特定ブロック

ラベルの有無に関係なく、どのスコープも参照可能。PROVしない雑多なメモのスコープも、実験ノートの試料タブも、同じように @ で指して別のノートから利用できる。

MOC (Map of Contents)

MOC は特別なノート種別ではない。@ 参照が密集したノートが自然に MOC として機能する。典型的な用途:

  • 研究ノート全体の俯瞰（「@実験1 の結果は @実験2 と整合、@文献A の報告を支持する」）
  • 複数実験の結果を集約した比較テーブルやチャート
  • 将来的には、@ 参照先のデータをインラインで引用・集約（トランスクルージョン）できる

ノートの種類はない — ラベル密度のグラデーション

ラベル密度典型的な内容動作
ラベルなし雑多なメモ、アイデア、日記Zettel のみ（@ 参照可能、PROV なし）
一部ラベル手順だけ構造化したメモZettel + 部分 PROV
フルラベル実験ノート（試料分割あり）Zettel + 完全 PROV + 試料タブ
@ 参照主体MOC、研究ノートまとめZettel 集約ビュー

メディアレジストリ: 画像・PDF も Zettel

ノートに挿入した画像やPDFは、そのノートの blocks[] に閉じ込められている。これでは「前に撮ったSEM画像どれだっけ」という横断検索ができない。同じ論文PDFを別ノートで使いたいときに再アップロードが必要になる。

メディアレジストリは、画像・PDF等のメディアファイルにノートとは独立した固有IDを付与し、ノート横断で管理する仕組み。Capacities のメディアデータベースに着想を得ている。

動作原理

  • アップロード時: ユーザーがエディタに画像をドロップすると、裏側でメディアレジストリに登録される（ユーザー体験は変わらない）。ファイルは Google Drive に独立保存され、ノート側の画像ブロックには mediaId（参照ID）だけを持つ。
  • 参照: 同じメディアを別のノートから @ で参照できる。実体は1つ、表示は複数。
  • PROV: メディアに #結果 や #属性 ラベルを付ければ、PROV Entity として来歴を追える（「このSEM画像は Cu焼結ノート/焼結/S-A の結果として生成された」）。

メディアの種類と実験科学での活用

種類具体例横断的に見る価値
画像SEM画像、光学顕微鏡写真、外観写真全実験のSEM画像をギャラリーで一覧表示 → 共通パターンの発見
グラフXRDパターン、応力-ひずみ曲線異なる実験のXRDを重ね合わせ比較
PDF参考文献、装置マニュアル1つの論文を複数実験ノートから参照（再アップロード不要）
データファイルCSV、Excel生データへの永続的リンク

「書く面は速い」の原則は変わらない。 ユーザーは画像をドラッグ&ドロップするだけ。メディアレジストリへの登録、固有IDの付与、Google Driveへの保存はすべて裏側で自動的に行われる。

7. # と @ — 2つのインライン記法

provnote のインライン操作は2つの記号に集約される。

# はラベル（PROV ロール）、@ は参照リンク（知識ネットワーク）。どちらもオートコンプリート付き。

# オートコンプリートの動作

# を打つとラベル候補のドロップダウンが出現する。1〜2文字でフィルタリングされ、Enter/Tab で確定。確定するとテキストから #xxx が消え、バッジが表示される。

3つの入口が等価: (1) # テキスト（主要な入口）、(2) スラッシュメニュー /手順（ブロック+ラベル同時生成）、(3) ドラッグハンドル（後付け）。どれも LabelStore.setLabel(blockId, label) を呼ぶだけ。

なぜ [] ではなく # か

[手順] は閉じ括弧・ラベル名・閉じ括弧の3要素が必要で打鍵コストが高い。# は1文字でトリガーになり、オートコンプリートで「# + 1文字 + Enter」の3打鍵で完結する。Slack / Obsidian で広く知られたパターンで学習コストが低い。

@ 参照の3つの入口

「あらゆるスコープが Zettel」「メディアも Zettel」というと、参照可能な対象は膨大になる。しかし全件をドロップダウンに出す必要はない。アドレス可能（技術的に指せる）と発見可能（候補に出る）は別の概念であり、候補の出し方を工夫すれば爆発しない。

入口1: ドリルダウン（場所を覚えている場合）

@Cu焼結/ と打つとそのノートの見出し一覧が出る。選ぶと試料タブ一覧 → ラベル付きブロック → と階層を辿れる。各段階で候補は数件〜数十件。最近触ったノートや同ノート内の参照に有効。

入口2: @ 後キーワード検索（場所を忘れた場合）

@SEM 800 のようにパスではなくキーワードを入力すると、全ノート横断でラベル・ファイル名・テキスト・メディア名を検索し、結果を同じドロップダウン内に表示する。ドリルダウンと検索は同じ入力欄で共存する（/ を含めばドリルダウン、含まなければ検索）。エディタを離れずにインラインで完結する。

入口3: ブラウズ → リンクコピー → ペースト（じっくり探す場合）

メディアギャラリーや別ノートで見に行き、目的のブロックやメディアの「リンクをコピー」→ 元のノートにペースト → 自動的に @ メンションに変換。コンテキストスイッチは発生するが、画像を一覧ブラウズして偶発的発見を得る場面や、特定のPDFを探す場面で有効。

候補に出る対象のフィルタリング

対象候補に出るか理由
ノートタイトル / H2・H3 見出し常時スコープの「入口」である。最も参照される単位
試料タブ名ドリルダウン時親スコープを選んだ後に出現
ラベル付きブロックドリルダウン時 + 検索ラベルで名前がつく = 発見可能にする意味がある
メディア（画像・PDF）ドリルダウン時 + 検索ファイル名・ラベルで引ける
ラベルなしの普通の段落出ないテキスト断片が並んでも選びようがない。リンクコピー(入口3)でのみアドレス取得可

リンクタイプは "reference" の1種類のみ。ユーザーにタイプ選択は求めない（Zettelkasten の「リンクはリンク」原則）。「支持する」「矛盾する」はリンク周辺のテキストが伝える。

8. ラベルシステム

コアラベル

入力例内部名バッジ表示PROV
#手[手順]手順prov:Activity
#使[使用したもの]使用したものprov:Entity + used
#結[結果]結果prov:Entity + wasGeneratedBy
#属[属性]属性prov:Entity + 親への関係

「使用したもの」が「材料」ではない理由: 材料（Cu粉末）だけでなく、装置（SPS装置）やソフトウェア（ImageJ）も手順の入力になる。PROV の prov:used で紐付く全般を包含する名前。

エイリアス

ALIAS_MAP により、ユーザーは好きな呼び方でラベルを付けられる。#材料 → [使用したもの]、#パターン → [試料] 等。オートコンプリートの検索ソースにもなる。

フリーラベル

コアラベルにもエイリアスにも一致しないラベル（#目的、#考察 等）はフリーラベルとして登録される。PROV には変換されないが、バッジは表示され、知識ネットワーク上で見出しの意味を示す。

ラベルの自動継承

箇条書きで Enter → 次行に同じラベル。ラベル付き箇条書きをインデント → [属性] に変化。

10. 試料とスコープ

試料はノート（実験）レベルの概念

試料テーブルは特殊なブロックではない。 普通のテーブルに #試料 ラベルを付けたことで分割定義として機能する。ラベルを外せば普通のテーブルに戻る。

ノート冒頭の [試料] テーブルが全手順の分割を定義する。各手順に個別のテーブルは不要。

sampleScope タブ

[試料] テーブルがノート内に存在すると、すべての [手順] スコープ内に試料タブバーが自動出現する。

## 焼結 #手順
使用したもの: Cu粉末 1g          → 共通（タブの外）
属性: 昇温速度 5℃/min           → 共通（タブの外）

──[S-A]─[S-B]─[S-C]───比較...──  → タブバー
│ S-A (800℃)                   │  → アクティブタブの内容
│ #結果 密度: 8.2 g/cm³         │
│ #属性 SEM画像: (画像)          │
──────────────────────────────

比較テーブル / チャート            → 共通（タブの外）
[考察] 温度の上昇に伴い...        → 共通（タブの外）

共通スコープ部分の活用

タブの外（共通スコープ部分）には、全試料を横断するコンテンツを配置できる:

  • 比較テーブル: 各試料の数値結果をまとめた表
  • チャート: 温度 vs 密度の散布図、条件ごとの棒グラフなど
  • 考察: 全試料を通した分析、議論
  • 共通操作メモ: 試料に依存しない記録

手順間の試料フロー

全 [手順] が同じ試料セットを参照する。試料の脱落はタブを削除せず「スキップマーク」を付けて理由を記録する（PROV 上は prov:wasInvalidatedBy）。

タブ連動モード（オプション）

デフォルト: 各手順のタブは独立（手順中心の視点）。連動モード ON: 1つ切り替えると全手順が同じ試料に（試料中心の視点 = 「S-A の旅を追う」）。

比較モード

タブバーの「比較」ボタンで2つのタブを横並び表示。主にリッチデータ（画像等）の比較に使う。

分割の中の分割は許容しない

多因子の条件（素材2種 × 温度3段階）はテーブルの列でフラットに表現する。PROV-DM の Activity にネストの概念はない。

スキャフォールドと補完性インジケーター

2つ目以降のタブに書き始めたときに初めて出現（ステージ4）。最初のタブだけ書いているうちは何も出ない。

  • スキャフォールド: 最初のタブの構造を空タブにプレースホルダー表示。クリックで実体化、無視も可。
  • 補完性インジケーター: タブバーに「S-B: 密度 ✓ SEM ✓ 外観 ✗」のような進捗表示。

なぜレーンではなくタブか

横並びレーンは3〜4試料で画面幅が破綻し、かつ編集もしにくい。タブ方式は試料数に制限なく、フル幅で編集可能。「記録は試料ごとに分かれている」「比較はテーブル/チャートで対応」という実感（前提1, 2, 3）に合致する。さらに共通コンテキストが上に残るので、ページ切替にともなうコンテキスト断絶が起きない。

試料タブの中身は Zettel

各試料タブの内容は独立した Zettel として @ で参照可能。別のノートから @Cu焼結ノート/焼結/S-A のように特定の試料の記録を引用できる。

11. PROV 構造化

漸進的構造化モデル

同じラベルでも、入力形式（ブロック型）によって PROV 出力の精度が変わる。

入力形式PROV 出力レベル
段落 Cu粉末 1gEntity { rdfs:label: "Cu粉末 1g" }メモ書き
テーブル 名前|量|純度Entity { rdfs:label: "Cu粉末", provnote:量: "1g" }構造化
画像ブロックEntity { rdfs:label: "SEM画像", prov:type: "Image" }メタデータ

key:value のパース

テーブルなら確実にパース（ヘッダー = key、セル = value）。自由テキストの key: value パターンは generator.ts 内で最善努力パースするが、ユーザーには知らせない、強制はしない（前提5）。

[属性] の PROV 化

prov:Entity として出力。親 Entity/Activity への関係で紐付く。画像・音声・自由記述を扱える。

PROV-JSON-LD 準拠出力

独自構造を W3C 標準の PROV-JSON-LD に置換。provnote: 名前空間でユーザー定義プロパティを格納（制限なし）。テーブルのヘッダーが provnote:量, provnote:純度 になる。

名前空間

名前空間用途制約
prov:W3C 標準の型・関係固定（変更不可）
provnote:ユーザー定義プロパティ（テーブルヘッダーから自動生成）制限なし
rdfs:label, comment標準

12. AI チャット

スコープ紐付け

AIとの対話は見出しスコープに紐づいて保存される（ScopeChat）。右パネルの Chat タブで継続対話。見出しブロックに 💬 インジケーターが表示される（チャットがある場合のみ）。

回答の活用

  • 「ノートに反映」→ AI の回答をブロックとしてスコープ内に挿入
  • 「別ノートとして派生」→ AI の回答を新しいノートファイルに変換

構造化提案の原則

AIによる構造化提案はチャットの中でのみ行い、エディタ上では割り込まない（前提5）。ユーザーが自分でチャットを開いて「もう少し整理して」と考えたときに、初めて AI が「テーブルにすると比較しやすくなりますよ」と返す。

13. 派生とコンテキスト保持

Split View

派生ノート作成時、右パネルが一時的に派生元ノートの読み取り専用表示に切り替わる。

パンくずリスト

派生チェーンの全パスを常時表示: 元ノート → ## 結果 → 🤖 派生ノート (現在)

sampleScope タブとの共通原理

ノート派生のコンテキスト断絶と、試料タブの切替時のコンテキスト断絶は、同じ問題の異なるスケール。どちらも「共通コンテキストを上に残す」ことで解決する。

UI の複雑さはユーザーが書いた量に比例する。

15. 右パネルと可視化 —「読む面は賢く」

設計原則の「書く面は速く、読む面は賢く」において、「読む面」は主に右パネルの4つのタブとエディタ上のインジケーターで実現される。

右パネルの4タブ

Graph ネットワークグラフ

現在のノートを中心に、@ 参照（知識リンク）とノート間派生（PROV リンク）で繋がるノート群を可視化。Cytoscape.js + fcose レイアウト。2ホップ BFS で近隣ノートを取得、グラフに表示する。

PROV 来歴グラフ

現在のノート内のラベル付きブロックから生成された PROV グラフを可視化。Activity（手順）→ Entity（使用したもの・結果）の因果関係を DAG として表示。Cytoscape.js + ELK レイアウト。試料分割がある場合は試料ごとの Activity が並列に表示される。

Chat AI 対話

見出しスコープに紐づく AI チャット。継続的な複数ターン対話。「ノートに反映」or「別ノートとして派生」の選択肢。スコープを選択すると、そのスコープに紐づく ScopeChat が表示される。

派生元 読み取り専用

派生ノート作成時のみ出現。派生元ノートの全体を読み取り専用で表示。メインエディタで派生先を編集しながら、右パネルで元のノートを参照できる。

ネットワークグラフの詳細

Graph タブは provnote の Zettelkasten 基盤を可視化するビュー。

ノードの種類

ノードソース表示
現在のノートアクティブなファイル中心に配置、ハイライト
@ 参照先ノートknowledgeLinks の targetNoteId1ホップ隣接
派生元・派生先derivedFromNoteId / provLinks1ホップ隣接、矢印付き
2ホップ先隣接ノートの knowledgeLinks薄く表示

エッジの種類

エッジソース視覚的表現
@ 参照knowledgeLinks (reference)破線、色なし（無色透明なリンク）
派生provLinks (derived_from)実線、矢印付き、方向あり

MOC 的なノート（@ 参照が密集したノート）はグラフ上で自然にハブ（多くのエッジが集まるノード）として浮かび上がる。特別なマーキングは不要 — トポロジーが語る。

ノードのクリック

グラフ上のノードをクリックすると、そのノートを開く。右クリックで「新しいタブで開く」「Split View で開く」等のコンテキストメニュー。

PROV グラフの詳細

PROV タブはノート内の PROV 構造を DAG として可視化するビュー。

ノード

  • Activity（手順）: [手順] ラベル付きスコープから生成。長方形、青系。
  • Entity（使用したもの・結果・属性）: 対応するラベル付きブロックから生成。楕円、黄系。

エッジ

  • prov:used → Entity → Activity（その材料をその手順が使った）
  • prov:wasGeneratedBy → Entity → Activity（その結果はその手順から生まれた）
  • prov:wasInformedBy → Activity → Activity（その手順は前の手順の結果を使った）

試料分割の表示

試料テーブルが存在する場合、1つの [手順] スコープから複数の Activity ノード（S-A用、S-B用...）が生成される。これらは並列に配置され、試料ごとの因果チェーンを視覚的に追える。

エディタ上のインジケーター

エディタ本体は「速い」を保ちつつ、以下の控えめなインジケーターでリンクやチャットの存在を知らせる。

インジケーター表示条件動作
🏷 ラベルバッジラベルが付いたブロッククリックでラベル変更・削除
💬 チャットマークScopeChat が存在するスコープの見出しクリックで右パネル Chat タブを開く
🔗 派生バッジ派生元 or 派生先がある場合クリックで派生先ノートを開く
@ インラインリンクテキスト内の @ 参照クリックで参照先にジャンプ

backlink の表示

あるスコープが他のノートから @ で参照されている場合、その参照元（backlink）を表示する必要がある。

  • 表示場所: 右パネルの Graph タブ下部に backlink 一覧セクション。
  • 内容: 参照元のノート名、スコープ名、参照を含むテキストの抜粋。
  • クリック: 参照元ノートを開き、該当ブロックにスクロール。
  • 自動生成: @ 参照を作成すると、ターゲット側に backlink が自動的に追加される。データとしては knowledgeLinks に逆方向の reference リンクを自動挿入する形になる。

メディアギャラリービュー

メディアレジストリに登録されたすべてのメディアを、ノートをまたいで一覧表示するビュー。左パネルのファイル一覧と同階層に「メディア」タブとして配置する。

  • 表示形式: グリッド形式のサムネイル一覧。画像はプレビュー、PDFはアイコン + ファイル名。
  • フィルタ: ファイル種別（画像 / PDF / データ）、ラベル（#結果 / #属性）、ノート別、日付順。
  • 操作: クリックでプレビュー、「リンクをコピー」で内部リンクをクリップボード、ノートにドラッグ&ドロップでエディタにメンション挿入。
  • 来歴表示: メディアを選択すると、PROV 上のソース（「このSEM画像は Cu焼結ノート/焼結/S-A の結果として生成された」）が表示される。
  • 活用場面: 全実験のSEM画像を一覧で眺めて共通パターンに気づく、特定の論文PDFを探して別ノートに引用する。

16. データモデル

3層構造

ProvNoteDocument

type ProvNoteDocument = {
  version: 2;
  title: string;
  pages: ProvNotePage[];
  noteLinks?: NoteLink[];
  derivedFromNoteId?: string;
  derivedFromBlockId?: string;
  generatedBy?: AgentMeta;
  chats?: ScopeChat[];
  createdAt: string;
  modifiedAt: string;
};

ProvNotePage

type ProvNotePage = {
  id: string;
  title: string;
  blocks: Block[];
  labels: Record<string, string>;
  provLinks: BlockLink[];
  knowledgeLinks: BlockLink[];
};

BlockLink

type BlockLink = {
  id: string;
  sourceBlockId: string;
  targetBlockId: string;
  type: ProvLinkType | "reference";
  layer: "prov" | "knowledge";
  createdBy: "human" | "ai" | "system";
  targetPageId?: string;
  targetNoteId?: string;
};

ScopeChat

type ScopeChat = {
  id: string;
  scopeBlockId: string;
  scopeType: "heading" | "block";
  messages: { role: "user"|"assistant"; content: string; timestamp: string }[];
  generatedBy?: AgentMeta;
  createdAt: string;
  modifiedAt: string;
};

課題解決策セクション
U1 コンテキスト断絶Split View + パンくずリスト。sampleScope タブの「共通コンテキストを上に残す」原理も同じ。§13, §10
U2 AIチャット散逸ScopeChat でスコープに紐づいて保存。右パネル Chat タブで継続対話。§12

P1 Entity 非構造化漸進的構造化モデル。テーブルで書けば構造化、テキストのままでも動作。§11
