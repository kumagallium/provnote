# 応用例: 実験科学 — 試料管理と3層構造

> このドキュメントは、Graphium の汎用的な仕組み（`@` 参照 + `#` ラベル + MOC + インデックステーブル）を**実験科学**に適用した場合のパターンを記述する。同様の構造は他のドメイン（プロジェクト管理のタスク追跡、ソフトウェア開発のリリース管理等）にも応用できる。

複数試料の比較実験を、`@` 参照 + `#` ラベルだけで管理する。新しいブロック型は不要。

---

## 実験の3層構造

実験の活動は3つの層に分かれる。これは TSJ2024 ポスターで提案した Plan Graph / Provenance Graph の構造を発展させたもので、MaiML フォーマットの多層モデルとも整合する。

| 層 | 役割 | Graphium での実現 | MaiML との対応 |
|---|---|---|---|
| **手順テンプレート** | 再利用可能な標準手順。「焼結はこうやる」 | 普通のノート。複数の実験から `@` で参照される | Protocol |
| **実験概要 (MOC)** | 1回の実験の計画。「この3条件を比較する」 | 普通のノート。インデックステーブルで条件定義 | Instruction |
| **試料ノート** | 各試料の実際の記録。「S-A で何が起きたか」 | 普通のノート。全5コアラベル（`#手順` `#材料` `#ツール` `#結果` `#属性`）を使って記録 | Result |

3つとも `GraphiumDocument`。データ型は同じ。`@` 参照とラベルの密度で「役割」が自然に決まるだけであり、特別なデータ構造は不要。これは統一モデル（Zettelkasten + オプショナル PROV）の帰結であり、「ノートの種類はない、ラベル密度のグラデーションだけ」という原則がここでも効いている。

> **補足:** 手順テンプレートは、インデックステーブルを持たない MOC ノート。実験概要ノートとの違いは、特定の条件（温度、粒径等）や特定の試料（`@S-A` 等）を含まず、汎用的な手順だけを記述している点。実験概要ノートが `@手順テンプレート` で参照することで、テンプレートが再利用される。つまり2つは別の種類ではなく、具体度の違い。

### Class と Instance の関係

この3層は Class/Instance の入れ子として理解できる。

```
手順テンプレートノート (Class)
  ↓ @ 参照でインスタンス化
実験概要ノート (Instance of Template / Class of Samples)
  ↓ @ 参照でインスタンス化
試料ノート (Instance of Instance)
```

手順テンプレートは複数の実験から使い回せる（Class）。実験概要はテンプレートのインスタンスであると同時に、試料ノート群に対する Class でもある。試料ノートは最も具体的なインスタンス。

### テンプレートの再利用

手順テンプレートが「使い回される Class」になるのは、複数の実験概要ノートから `@` で参照されているだけ。特別な仕組みは不要。

```
「焼結の標準手順」ノート
  ├─ ## 秤量 #手順
  │   Cu粉末 #材料
  ├─ ## 焼結 #手順
  │   SPS装置 #ツール
  │   昇温速度: 5℃/min
  └─ ## アニール #手順
      電気炉 #ツール

実験A 概要ノート → @焼結の標準手順 + インデックステーブル（温度3条件）
実験B 概要ノート → @焼結の標準手順 + インデックステーブル（粒径2条件）
  ↑ 同じテンプレートを半年後に別の実験で再利用
```

手順テンプレートを改良したら、参照しているすべてのノートから最新版が見える。`derivedFromNoteId`（派生リンク）を使えば「このノートはあのテンプレートから派生した」という PROV 上の来歴も残る。

## 具体的な構造

```
手順テンプレートノート (Protocol)
├─ ## 焼結の標準手順 #手順
│   SPS装置 #ツール
│   昇温速度: 5℃/min
└─ ## アニールの標準手順 #手順
    電気炉 #ツール

実験概要ノート (MOC = Instruction)
├─ 目的: Al3Vの単相作成のため、アニーリングの有無を確認する
├─ 手順: @焼結の標準手順
├─ インデックステーブル:
│   │ 試料        │ 組成    │ 温度    │ 密度  │ 状態  │
│   │ @S-A ノート  │ Al3V   │ 800℃   │ 8.2   │ 完了  │
│   │ @S-B ノート  │ Al3V   │ 900℃   │ 8.7   │ 完了  │
│   │ @S-C ノート  │ Al3.1V │ 1000℃  │ —     │ 破損  │
├─ 比較チャート: 温度 vs 密度
├─ 画像ギャラリー: 各試料の SEM 画像を @参照で横並び表示
└─ 考察: 温度の上昇に伴い密度が向上。S-Cは昇温速度の見直しが必要。

S-A ノート (Zettel = Result)
├─ ## 秤量 #手順
│   Cu粉末 1g #材料
│   電子天秤 #ツール
├─ ## 焼結 #手順
│   800℃, 2h
│   密度: 8.2 g/cm³ #結果
│   SEM画像 #属性
└─ ## アニール #手順
    500℃, 1h
    密度変化なし #結果
```

## ユーザー操作の流れ

1. 手順テンプレートノートを作成（既存があれば再利用）
2. 実験概要ノートを作成し、目的を書き、`@手順テンプレート` で手順を参照
3. インデックステーブル（`/インデックステーブル`）を作成し、各行に試料名・条件を記入
4. テーブルから試料ノートを一括作成（テンプレート適用可能）
5. 各試料ノートを開いて実験を記録（通常のノート操作。D&D、カラム等すべて使える）
6. 結果が出たら概要ノートに戻り、比較テーブル・チャート・考察を記入
7. 概要ノートのテーブル行をクリック → 試料ノートをピーク表示で確認

## sampleScope タブ: 検討と廃止の経緯

Graphium v1 では `sampleScope` というカスタムブロック型を導入し、1つのノート内にタブ UI を作って試料ごとの記録を切り替える設計を検討・プロトタイプした。

### sampleScope の設計概要

- `sampleScope` は BlockNote のカスタムブロック型として定義
- ノート内に インデックステーブルがあると、各 `#手順` スコープ内に sampleScope ブロックが自動挿入される
- sampleScope の中は試料ごとにタブで切り替え。タブの外が全試料共通のコンテンツ（材料、条件、考察等）
- 各試料タブの内容は、ネストされた BlockNote エディタインスタンスとして独立して動作（案A）
- データは `props.samples` に試料ごとの `Block[]` を JSON 文字列として格納

### 検討した実装方式

**案A（ネストされたエディタ）:** sampleScope 内に試料ごとの独立した BlockNote エディタインスタンスを配置。プロトタイプまで実装済み。データ分離は明確だが、異なるエディタインスタンス間のドラッグ&ドロップが BlockNote 標準で未対応。タブ内ではブロックハンドルやマルチカラムが使えるが、タブ境界をまたぐ移動はカット&ペーストのみ。

**案B（単一エディタ + フィルタ表示）:** sampleScope の children にすべての試料のブロックを入れ、各ブロックに `sampleId` prop を持たせて表示フィルタで切り替える方式。ドラッグ&ドロップがネイティブで動く利点があるが、BlockNote が children の一部を非表示にする挙動を許容するか未検証。プロトタイプは作成していない。

### 廃止の理由

| 問題 | 詳細 |
|---|---|
| 実装の複雑さ | カスタムブロック型、ネストされたエディタ（or フィルタ表示）、props.samples のJSON文字列管理 |
| D&D 制約 | 案A ではタブ境界をまたぐドラッグ不可。案B は未検証。 |
| 認知負荷 | 共通スコープとタブスコープの境界を常に意識する必要がある |
| 設計思想との矛盾 | 「Zettelkasten が基盤」と言いつつ、試料管理だけ独自のブロック型が必要だった |
| 元の直感との乖離 | TSJ2024 ポスターでは研究テーマページと実験ページを別ページに分離していた。sampleScope はこの自然な分離を1ノートに押し込もうとしていた |
| 3層構造と不整合 | 手順テンプレートの再利用（Protocol 層）が sampleScope では表現しにくかった |

3層分離（手順テンプレート + 実験概要 + 試料ノート）は、Phase 1-2 の `@` 参照 + MOC パターンに加え、インデックステーブル（行がノートを表すシンプルなカスタムブロック型）だけで動作する。sampleScope のような複雑なカスタムブロック（ネストエディタ、タブ UI）は不要。sampleScope のプロトタイプは `feature/sample-scope-v1` ブランチに保存してあり、将来的に再検討する場合に参照可能。

## 将来的な拡張: MOC 側での集約

実験概要ノート（MOC = Instruction）が、各試料ノート（Result）から結果を自動的に集約できると、Instruction と Result の橋渡しが強力になる。

### トランスクルージョン（自動集約）

MOC の条件テーブルの結果列が、各試料ノートの `#結果` ラベル付きブロックから値を自動的に引用する。

```
概要ノートのテーブル:
│ 試料   │ 温度   │ 密度              │
│ @S-A   │ 800℃  │ ← @S-A/焼結/密度  │  ← S-A ノートで「8.2」と書くと
│ @S-B   │ 900℃  │ ← @S-B/焼結/密度  │     ここに自動反映される
```

これにより、研究者は各試料ノートに結果を書くだけで、概要ノートの比較テーブルが自動的に埋まっていく。手動でコピーする必要がなくなる。

### 画像ギャラリーの自動構成

MOC に「各試料の SEM 画像を並べたい」と指定すると、各試料ノートの `#属性` ラベル付き SEM 画像を `@` 参照で自動収集し、グリッド表示する。

### チャートの自動生成

条件テーブルの数値列からチャート（棒グラフ、散布図等）を自動生成。トランスクルージョンで値が更新されれば、チャートも連動する。

これらの集約機能はすべて MOC 側（実験概要ノート）で行われる。試料ノート側は普通のノートのまま何も変わらない。「書く面は静か」の原則を維持しつつ、「読む面（MOC）が賢い」を実現する。
