# 設計原則・ドメイン前提

すべての設計判断はここから導かれる。

---

## 設計原則

### 書く面は静か、読む面が賢い

エディタは普通のメモ帳のまま。ユーザーの行為（ラベル付与、参照作成など）が機能を「召喚」する。何も書かなければ何も出ない。

### ブロックは素材、ラベルは意味

ラベルはブロック型と独立。H2 でも段落でもテーブルでも、同じラベルを付けられる。ブロック型は「素材」、ラベルは「意味」、PROV は「計算結果のビュー」。

### PROV は編集対象ではなく、計算結果のビュー

PROV-JSON-LD は `labels` + `provLinks` から毎回生成されるリードオンリーのビューであり、保存しない。ユーザーが PROV グラフを直接操作する場面はない。

### 各フェーズは独立して価値がある

Phase 1 だけでもリリース可能。後のフェーズを作らない判断をしても、それまでに作ったものは無駄にならない。

## ターゲットとプログレッシブ・ディスクロージャー

Graphium は**汎用ノートアプリ**として位置づける。あらゆる分野の作業記録・思考整理・プロジェクト管理に使える。ただし、その構造は2つの層に分かれている。

**基盤層: Zettelkasten ベースの汎用ノート。** `@` 参照、backlink、ネットワークグラフ — どんな分野のメモや思考整理にも使える。Obsidian や Notion のユーザーが「@ メンション」として知っている操作感と同じ。ラベルを一切付けなくても、この基盤だけで Zettelkasten として十分に価値がある。

**拡張層: PROV 来歴追跡。** `#手順` `#Input` `#Output` 等のラベルを付けると、来歴グラフ（PROV-DM）が自動生成される。「何を入力に、どんな手順で、何が出力されたか」というプロセスの追跡はドメインを問わず有用であり、実験科学・ソフトウェア開発・プロジェクト管理・製造工程など幅広い分野で活用できる。コアラベルの表示名は汎用的（Input / Output / Tool）だが、エイリアスにより各分野の用語（材料・試薬・結果など）でも入力可能。

この構造は**プログレッシブ・ディスクロージャー**（段階的開示）として設計されている。ユーザーが何もしなければ普通のメモ帳。ラベルを付け始めると PROV が出現する。インデックステーブルを作れば管理の構造が使える。高度な機能は、ユーザーがそれを必要とする行為をしたときにだけ姿を見せる。

## ドメイン応用例: 実験科学

Graphium は汎用ノートだが、設計の検証と発展は実験科学をリードドメインとして行っている。以下は実験科学の現場の実態から導かれた前提であり、このドメインでの活用パターンの土台となる。他のドメイン（ソフトウェア開発、プロジェクト管理等）でも「手順 → Input → Output」の構造は同様に適用できる。

### 前提1: 比較したいのは条件と結果、記録は試料ごとに分かれる

研究者が横並びで見たいのは数値データ（テーブル/チャートで対応）。構造化データの比較テーブルやチャートは実験概要ノート（MOC）に置く。

**注意:** 生物学のウェスタンブロットのように「バンドの濃さを目視で比較する」場合は、テーブルでは不十分で画像の横並びが必要。この種のリッチデータ比較には、MOCノートに画像ギャラリーブロック（複数の `@` 参照画像をグリッド表示）を配置する形が理想的。

### 前提2: 記録は試料ごとに分かれていることが多い

典型的には1つずつ順番に処理して記録する。ただし複数試料を同時に処理できる装置もある（多チャンネル測定器、バッチ式熱処理炉）。この場合でも記録は試料ごとに分かれていることが多い。全試料共通の操作メモは概要ノートに書ける。

**注意:** 高スループットスクリーニング（HTS）のように、96ウェルプレートで一度に多数の試料を処理する分野では、試料ごとに個別ノートを作るのは非現実的。この場合は抽象度を上げて「1プレート = 1ノート + 結果データファイル（CSV/Excel）を `#結果` メディアとして添付」という粒度で扱う。各ウェルの結果はデータファイル内の行であり、Graphium のデータ可視化機能でヒートマップ等として表示する。「1試料 = 1ノート」は推奨パターンであって強制ではなく、粒度の選択はユーザーに委ねられる。

### 前提3: 比較には構造化データとリッチデータの2種類がある

**構造化データの比較**（数値結果）はテーブルの列とチャート（棒グラフ、散布図等）で完結する。これらは実験概要ノート（MOC）に置く。

**リッチデータの比較**（SEM画像の横並び、XRDパターンの重ね合わせ等）はテーブルには入らない。MOCノートに画像ギャラリーブロック（複数の `@` 参照メディアをグリッド/横並び表示するブロック）を配置して対応する。各メディアはメディアレジストリから `@` で参照するため、実体は各試料ノートに1つだけ存在し、MOCでは参照として並べる。

### 前提4: 試料セットは実験の最初に決まる

研究者は実験の最初に「今回の試料は S-A, S-B, S-C の3つ」と計画する。全手順を通じて共通。

**注意:** 探索的な研究では「やってみてから次の条件を決める」こともあり、試料セットが途中で拡張されることもある。この場合は概要ノートのテーブルに行を追加し、新しい試料ノートを作成すればよい。

### 前提5: key:value の入力を強制できない

急いでメモすることもある。漸進的構造化（テキストのままでも動く → テーブルで書けば構造化される）。構造化の提案はAIチャット内のみ。

### 前提6: 試料間で key を揃える価値はあるが、強制はできない

揃えば比較テーブルやチャートの自動生成に使える。テンプレートで動機づけするが強制はしない。

**注意:** 「S-A は密度を測ったが S-B は測れなかった」は現実に起こる。欠損を許容する設計が必要。

## 用語集

### コアラベル

| 表示名 (en) | 表示名 (ja) | 内部名 | PROV | 意味 |
|---|---|---|---|---|
| Procedure | 手順 | `[手順]` | prov:Activity | プロセスの1ステップ |
| Input | インプット | `[材料]` | prov:Entity (used, material) | 手順への入力 |
| Tool | ツール | `[ツール]` | prov:Entity (used, tool) | 手順で使う道具・ソフト |
| Attribute | 属性 | `[属性]` | prov:Entity property | ブロックの補足情報 |
| Output | アウトプット | `[結果]` | prov:Entity (wasGeneratedBy) | 手順の出力 |

> **内部名と表示名の関係:** 内部名（`[材料]`、`[結果]` 等）は実験科学をリードドメインとして開発した経緯から残っている。ユーザー向け表示名は i18n により汎用的（Input / Output）に表示される。エイリアスにより、各ドメインの用語（材料・試薬・Reagents 等）でも入力可能。

### 一般用語

| 用語 | 意味 | Graphium での対応 |
|---|---|---|
| PROV-DM | W3C 標準の来歴データモデル | generator.ts が出力 |
| Provenance Graph | プロセスの来歴（何が起きたか） | # ラベルから自動生成 |
| MOC (Map of Contents) | 複数ノートを集約する概要ノート | `@` 参照が密集したノートが自然に MOC として機能 |

### 実験科学での対応

| 実験科学の用語 | Graphium の対応 |
|---|---|
| 試料 (Sample) | 試料ノート（Zettel） |
| 手順テンプレート (Protocol) | 再利用可能な手順ノート |
| 実験概要 (Instruction) | MOC ノート（インデックステーブル付き） |
| Plan Graph | 実験概要ノート |
| Provenance Graph | 試料ノートの # ラベルから自動生成 |
