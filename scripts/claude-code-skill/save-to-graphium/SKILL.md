# Save to Graphium

会話の流れを Graphium のノートとして保存するスキル。ローカルの Graphium ノート一覧（`~/Documents/Graphium/notes/`）に書き込み、ユーザーが Graphium を再読み込みすれば通常のノートと同じように表示される。Inbox 等の staging は持たない。

**重要**: 1 会話 = 1 ノートとは限らない。議論が複数の独立したトピックにまたがる場合は、**トピックごとにノートを分割して、スキルを複数回呼び出す**（「トピック分割の判断」セクション参照）。

## いつ使うか

- 「この議論を Graphium に保存して」「Graphium にまとめて」と指示されたとき
- 会話内の設計判断・調査結果・手順などをプロジェクト外に残したいとき
- ユーザーが明示的に `/save-to-graphium` を呼び出したとき

## やること

1. **議論から「問い → 答え」のペアを抽出する**
   - このスキルは **1 つの問い（仮説・疑問）と、それに対する答え（検証・結論）= 1 ノート** を基本単位とする
   - 1 会話に複数の独立した問いがあれば、**問いの数だけノートを作る**（スクリプトを複数回呼ぶ）
   - 詳細は「ノートの単位 — 問い × 答え」セクション参照

2. **各ノートを下書きする（問いごとに繰り返す）**
   - **タイトル**: 1 行・60 字以内・日本語可。**命題形（A は B である）または問い形（なぜ A は B なのか）** で書く。「〜について」「〜に関する考察」「〜の話」のような説明形は避ける
     - ❌ 「Graphium 保存 Skill についての議論」「設計の話」
     - ✅ 「Graphium 保存 Skill は Inbox を持たず Drive に直接書く」「なぜ複数ノート分割を save 側に寄せたのか」
   - **本文**: Markdown。「ノートの単位 — 問い × 答え」の構造を土台に、短く具体的に
   - 会話で出た **キーワードや固有名詞は必ず残す**（後から検索でヒットするように）
   - **冒頭 1-2 行は答えを言い切る**: 「本ノートでは…を扱う」「この議論では…」のようなメタ要約は禁止。最初の一文で結論を出す
   - 複数ノートに分ける場合、各ノートの末尾に `## 関連ノート` セクションを設けて他ノート名を列挙する

   詳細な書き方は **「まとめ方ガイドライン」** セクション参照。

2. **スクリプトを呼び出して書き込む**

   このスキル自身のパス（例: `~/.claude/skills/save-to-graphium/save.mjs`）を `node` に渡し、JSON を stdin で流し込む:

   ```bash
   node ~/.claude/skills/save-to-graphium/save.mjs <<'EOF'
   {
     "title": "<タイトル>",
     "body": "<Markdown 本文>",
     "source": "<任意: Claude Code のセッションIDなど>",
     "model": "<自分のモデル ID。例: claude-opus-4-7>"
   }
   EOF
   ```

   - ヒアドキュメントの区切り (`EOF`) は本文と衝突しない文字列にすること
   - body の改行や `"` は JSON としてエスケープする
   - `model` には **自分（Claude）が知っている自分のモデル ID** を渡す。システムコンテキストの "The exact model ID is ..." を参照する
   - スクリプトは `noteId`、`filePath`、`author` を含む JSON を 1 行で返す
   - **会話コンテキストに含まれる userEmail を勝手に `author.email` に差し込まないこと**（プライバシー方針セクション参照）

3. **ユーザーに報告する**
   - 書き込んだファイルパスと noteId を伝える
   - 「Graphium を再起動（または再読み込み）するとノート一覧に表示されます」と添える
   - 本文の要約を 1-2 行で返し、望ましくなければ上書きできることを示す

## ノートの単位 — 問い × 答え

このスキルが書くノートは原則すべて、**1 つの問いと、それに対する答え** で構成される 1 枚のメモ。「議論の議事録」ではなく「結論の付箋」を作るイメージ。

### 構造（基本テンプレ）

```
（冒頭 1-2 行で答えを敬体で言い切る。見出しなし）

## 問い
（議論の出発点になった疑問・仮説を 1-3 文で。なぜそれが気になったかも一言）

## 答え
（検証・議論の結果見えてきたこと。固有名詞・数値・代替案と却下理由はここに残す）

## 残る不安・次の一手
（任意。なければ書かない）

## 関連ノート
（任意。複数ノートに分けたときだけ）
```

- セクションはこの 3〜4 つで十分。さらに増やさない
- **冒頭の 1-2 行は答えそのもの**を敬体で書く。「本ノートでは…」「この議論では…」のようなメタ要約は禁止
- **「答え」が薄いまま終わった議論** でも書いて OK。その場合は冒頭で「結論はまだ出ていません」と正直に書き、`残る不安` に何が引っかかっているかを残す
- コードブロック・テーブルは必要最小限。長いコードはファイルパス + 行番号で参照する

### ノートを分割する基準（問いの数で決める）

- 議論に **独立した問いが N 個** あれば → N ノート（スクリプトを N 回呼ぶ）
- 「問い A の答えを出すために問い B を経由した」だけなら → 1 ノート（B は議論の足跡として答えに含める）
- 雑談・本題から外れた寄り道で問いが立っていないなら → ノートにしない

判断の軸は「**問いが独立して再利用されうるか**」。後日「あの件どうだったっけ？」と検索したときに、その問いだけで意味が通るなら独立ノート。

### 1 会話あたりの典型

- ふつうの会話: 1〜2 ノート
- 1 時間級の濃い議論: 3〜4 ノート
- 5 ノートを超えるなら、問いを上位概念でまとめ直せないか再検討する

### 例外: 単純メモ

「TODO の種」「ふと思いついたメモ」のような、問い × 答えの構造に乗らない短い気づきは、見出しなしの 2-5 行のメモで OK。無理に問い形にしない。

### 相互参照

分割した各ノートの末尾に:

```
## 関連ノート

- 他のノートのタイトル1
- 他のノートのタイトル2
```

ノート名（タイトル）で参照する。Graphium 側で開いたあと `@ノート名` 記法に書き換えれば相互リンクになる。スキルから書く時点では単純にタイトルを列挙するだけで十分。

## まとめ方ガイドライン

ノートの質を揃えるための指針。**機械的に従う規則ではなく、判断の軸**として使う。

### 読者モデル（誰が後で読むか）

ノートは少なくとも 3 人の読者を想定する:

1. **後日の自分（今から数週間〜数ヶ月後）**: 会話の文脈は忘れているので、「何を検討して、何を決めたか」だけで意味が通る必要がある
2. **Graphium の Knowledge Layer Ingester (LLM)**: あとで「Add to Knowledge」を押すと、このノートから Summary や Concept が生成される素材になる。**固有名詞・数値・リンク** が失われると情報が残らない
3. **他のセッションの AI（`G-XSESSION` 経由で引き継ぎ）**: コンテキストとして注入される前提。会話の冗長部分は省いた方が注入効率が高い

### 残すもの / 削るもの

**必ず残す**:

- **決定とその根拠**（「なぜ A ではなく B にしたか」）
- **固有名詞**: ファイルパス、関数名、ライブラリ名、環境変数、エラーメッセージ
- **数値**: 閾値、バージョン番号、設定値、ベンチマーク結果
- **代替案と却下理由**（後から「なぜこの案にしなかったか」を再検討できるように）
- **未解決の TODO・Next Action**（日付や期限があれば含める）

**削ってよい**:

- 雑談・挨拶
- 「了解しました」「ありがとうございます」的な確認の応酬
- 誤回答を訂正した往復の過程（最終的な正答だけ残す）
- 試行錯誤のうち却下された分岐（決定的な分かれ目だけ残す）

### 機密情報の扱い

Graphium のノートは将来 `G-KPACK` (Knowledge Pack) で外部公開される可能性がある。デフォルトで以下は避ける:

- **絶対に書かない**: API キー、パスワード、トークン、証明書、個人情報（電話番号・住所）
- **書かないのが無難**: 社内識別子・顧客名・未公開プロジェクト名（ユーザーが明示したときだけ書く）
- **相対パスを優先**: `/Users/<name>/...` のような絶対パスは `~/...` またはプロジェクトルート相対にする
- **メールアドレス**: 自動では書かない（プライバシー方針セクション参照）

### 長さ

**短く書く**。1 ノートは 1 つの問い × 1 つの答えしか入らないので、自然と短くなるはず。

- 単純メモ: 数行〜200 字
- 通常の問い × 答え: 300〜800 字
- 議論の重い問い: 800〜1500 字
- これを超えそうなら、**問いを 2 つに分けられないか** 再検討する（分割が薄まらない範囲で）

### 文体・温度感

このスキルが書くノートは、後で本人や他のセッションの AI が読み返す **個人の議論ログ**。冷たい report 調にせず、読み返したくなる文体で書く。Graphium のブログ用 `graphium-writing` スキルの文体ルールをほぼ踏襲する。

#### 絶対ルール

- **敬体（ですます調）で統一**する。常体（だ／である）は使わない
- **強い語彙を避ける**: 「賭ける」「絶対に」「圧倒的に」「劇的に」「振り切った」のような盛った言葉は使わない。代わりに「選ぶ」「決める」「判断する」「採用する」など落ち着いた語彙を使う
- **断定を避け「私はこう考えています」のスタンス**で書く。「〜が正しい」「〜すべき」「〜に決まっている」のような断定的主張はしない。「〜を選びました」「〜と考えています」「〜と感じています」「〜と見ています」を使う
- **em dash（—）は本文で使わない**。日本語では一般的でないため、接続詞や読点で繋ぐ
- **体言止めは控えめに**。1 段落に何度も使わない

#### 推奨ルール

- **一文は 60〜90 字を目安に**。100 字を超えたら論理ステップで切れないか検討する。論理を 3 つ以上詰め込まない
- **文末バリエーション**: 「〜です」「〜と考えています」「〜と見ています」「〜のではないでしょうか」を使い分け、「〜ます。〜ます。」の連続が 3 文以上続かないようにする
- **主語は「私（ユーザー）」に立てる**。「私はこう考えました」「自分はこの形を選びました」のように一人称を立てる
- **自己弁護的な前置きを書かない**: 「〜と主張するつもりはありませんが」「〜が正解だと言いたいわけではなく」のような言い訳めいた前置きは書かない。一人称で書けば押し付けがましくないトーンは自然に伝わる

#### Bad / Good

❌ 冷たい report 調:
> 本ノートでは Graphium の保存機能における設計判断について議論する。複数のアプローチを比較した結果、Drive 直書き方式を採用することとした。

✅ 短く・具体的・温度あり・敬体:
> Inbox は持たず Drive に直接書くことにしました。Inbox を作ると同期タイミングのバグが増えますし、ユーザーから見ると「保存したのにどこ？」となるからです。

❌ 強すぎる断定・盛った語彙:
> Drive 直書きが絶対に正しい。Inbox 方式に賭けるのはリスクが大きすぎる。

✅ 落ち着いたスタンス:
> Drive 直書きを選びました。Inbox 方式はバグが増えそうで、自分の使い方には合わないと感じています。

両者は同じ事実を伝えていますが、後者は読み返したくなります。後者を目指します。

#### リズムの作り方（重要）

文体ルールが揃っていても、リズムが単調だと読み手は途中で離脱します。次の点を意識します。

**❌ リズムが悪い例（「ます。」連続・体言止め多用・論理を詰め込みすぎ）**:

> 設計判断について議論しました。複数のアプローチを比較しました。Drive 直書き方式を採用しました。Inbox 方式は同期バグの懸念があります。ユーザーへの見え方の問題もあります。これらの理由から不採用としました。

**✅ リズムを整えた例**:

> Drive 直書きを選びました。Inbox を作ると同期タイミングのバグが増えますし、ユーザーから見ると「保存したのにどこ？」になります。Inbox の利点（書き込みの即時性、後から整理できる）と比べても、自分の使い方には合わないと感じています。

**改善のポイント**:

1. **文末を交互に**: 「〜ます」「〜ました」「〜と感じています」「〜と考えています」「〜のではないでしょうか」を混ぜ、同じ語尾を 3 文以上続けない
2. **論理ステップで切る**: 1 文に 3 つ以上の論理を詰めない。逆接（「ただし」「とはいえ」）や理由（「なぜなら」「というのも」）は文頭に置いて新しい文を始める
3. **体言止めは 1 段落 1 回まで**: 「〜という結果。」「〜という判断。」を連発しない。接続詞で繋ぐか、文として完結させる
4. **一文 60〜90 字を中心に**: 100 字を超えたら切る位置を探す。短すぎる「〜です。」の 3 連続も避ける

#### 残す情報

- **ユーザーの本音の言葉** はそのまま引用する（「すっきりしない」「自信が欲しい」など）。言い換えると温度が飛ぶ
- **認識が変わった瞬間** が答えの核なら必ず残す（「A だと思っていたが B だった」）
- 冗長な往復（了解・了承・感謝の応酬）は削る

## 誰が保存したかの記録（プライバシー方針）

`save.mjs` は `generatedBy.user` にノートの作成者情報を記録する。プライバシーを考慮し、以下の方針で動く:

| 項目 | 扱い |
| ---- | ---- |
| `username` (OS ユーザー名、例: `masayakumagai`) | **常に記録される** (`os.userInfo().username`)。Graphium 内でどの PC から保存されたか識別するための最小情報 |
| `email` | **opt-in のみ**。下記いずれかの場合に記録される |

email が記録されるのは以下いずれかの場合だけ:

1. 環境変数 `GRAPHIUM_USER_EMAIL` が設定されているとき
2. 呼び出し JSON に `author.email` を明示的に渡したとき（例: ユーザーが「自分のメールアドレスも記録して」と指示した場合のみ）

**Claude は会話コンテキストに `userEmail` があっても、ユーザーの明示的な同意なしに `author.email` として渡してはいけない**。Graphium のノートは将来 Knowledge Pack (`G-KPACK`) などで外部公開される可能性があるため、デフォルトで email を書き残すとプライバシーリスクがある。

ユーザーが「誰が保存したかを Graphium のログイン email で残したい」と明示したときだけ、以下のように渡す:

```json
{ "title": "...", "body": "...", "author": { "email": "user@example.com" } }
```

## 保存先のカスタマイズ

保存先は以下の優先順で解決される:

1. 環境変数 `GRAPHIUM_NOTES_DIR`（一時的な上書き用途）
2. Graphium 本体の設定ファイル
   (`<OS app config>/com.graphium.app/config.json` の `graphiumRoot`)
   → 設定があれば `<graphiumRoot>/notes/` に書き込む
3. デフォルト `~/Documents/Graphium/notes/`

つまり、Graphium の設定画面（General タブ → Local save location）で保存先を変えれば、
このスキルも自動的に同じ場所に書き込むようになる。環境変数はテスト・一時切替用。

```bash
# 明示的に上書きしたいとき
GRAPHIUM_NOTES_DIR="/path/to/notes" node .../save.mjs <<EOF ...
```

## やらないこと

- **ファイルを自動で開く・ブラウザを起動する** — スキルは書き込みだけで完結する
- **Graphium のインデックスを直接書き換える** — 次回起動時に Graphium 側が差分更新する
- **画像・添付ファイルの書き込み** — テキストのみ。メディアは Graphium 側で追加する
- **無理に 1 ノートにまとめる** — 複数トピックは複数ノートに分割する（スクリプトは 1 呼び出し = 1 ノートなので、複数回呼ぶ）

## 注意

- 対応する Markdown:
  - **ブロック**: 見出し h1-h3 / 箇条書き (`-` `*` `1.`) / コードブロック / パラグラフ / **テーブル** （`| col | col |` + `|---|---|` セパレータ行必須）
  - **インライン装飾**: `**bold**` / `*italic*` / `` `code` `` / `[text](url)`
  - **未対応**: ネストされた箇条書きの階層保持、打ち消し線 `~~`、画像埋め込み `![alt](url)`、脚注。これらはテキストとしてそのまま残る
- 書き込み後に Graphium アプリが起動中でも、アプリ内の一覧は自動更新されない。ユーザーに再読み込みを促すこと
